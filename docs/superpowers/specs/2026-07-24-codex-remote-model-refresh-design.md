# Codex 원격 모델 새로고침 설계

## 목표

Neru의 Codex OAuth 설정에서 사용자가 모델 새로고침 버튼을 누를 때마다 현재 계정에 제공되는 Codex 모델 목록을 원격 서버에서 정확히 한 번 조회한다. 백그라운드 폴링이나 앱 시작 시 주기적 조회는 추가하지 않는다.

## 현재 문제

현재 `CodexDirectClient.listModels()`는 `@earendil-works/pi-ai`에 내장된 정적 모델 목록만 반환한다. 설치된 `pi-ai`의 목록에는 GPT-5.6 모델이 없어, 계정에 모델 사용 권한이 있어도 설정 화면에 표시되지 않는다.

## 선택한 접근

Electron main 프로세스가 기존 Codex OAuth 자격 증명으로 공식 Codex 모델 카탈로그 엔드포인트를 직접 호출한다.

- 요청 경로는 `GET https://chatgpt.com/backend-api/codex/models?client_version=<version>`을 사용한다.
- `client_version`은 Neru가 실제로 지원하는 Codex 모델 계약의 호환 버전 상수로 관리한다. 최초 값은 GPT-5.6 Codex 지원 최소 버전인 `0.144.0`으로 두며, 응답 계약 호환성을 검증하지 않은 채 임의로 높은 값을 보내지 않는다.
- 헤더에는 OAuth bearer 토큰과 토큰에서 형식을 검사해 추출한 `chatgpt-account-id`를 포함한다.
- renderer에는 토큰, 계정 ID, 원격 응답 원문을 전달하지 않는다.
- 기존 모델 새로고침 IPC 호출 하나가 원격 요청 하나를 발생시킨다.
- 자동 재시도, 타이머, 백그라운드 갱신은 추가하지 않는다.

`pi-ai` 의존성만 업데이트하는 방법은 카탈로그가 다시 낡을 수 있어 제외한다. Codex CLI 또는 app-server의 모델 목록을 사용하는 방법은 현재의 직접 OAuth 구조에 별도 프로세스 의존성을 다시 도입하므로 제외한다.

## 모델 변환과 사용

원격 응답의 `models` 배열을 런타임 모델로 정규화한다.

- `visibility`가 선택 목록용인 모델만 표시한다.
- 모델 ID는 `slug`, 표시 이름은 `display_name`을 사용한다.
- 추론 강도는 `supported_reasoning_levels`의 `effort` 값을 사용한다.
- 원격 모델은 기존 Codex Responses 전송 규약과 공급자 URL을 유지한다.
- 원격 응답에 필요한 로컬 런타임 메타데이터가 없으면 호환되는 내장 Codex 모델의 전송 메타데이터를 복사하되, ID·이름·추론 강도는 원격 값을 우선한다.
- 새로 선택한 원격 모델 ID는 설정 표시뿐 아니라 실제 `stream()` 요청에도 사용한다.

## 상태와 폴백

성공한 원격 목록은 현재 Electron 프로세스 메모리에만 보관한다. 디스크 캐시를 새로 만들지 않는다.

- 첫 원격 조회 전에는 기존 내장 목록을 사용할 수 있다.
- 원격 조회가 성공하면 해당 결과로 현재 목록을 교체한다.
- 빈 목록, 스키마 오류, 인증 오류 또는 네트워크 오류가 발생하면 기존의 마지막 정상 목록을 유지하고 호출자에게 오류를 반환한다.
- 인증 만료 시 기존 자격 증명 갱신 경로를 한 번 거친 뒤 모델 조회를 수행하되, 모델 조회 자체를 자동으로 반복하지 않는다.
- 로그에는 상태 코드와 요약 오류만 남기고 토큰, 계정 ID, 응답 본문은 남기지 않는다.

## 보안 경계

- OAuth 자격 증명 접근과 네트워크 요청은 Electron main 프로세스에 한정한다.
- JWT payload는 서명 검증 수단이 아니라 이미 인증된 액세스 토큰에서 계정 라우팅 헤더를 구성하는 용도로만 읽는다.
- 원격 응답은 신뢰하지 않고 필요한 필드와 허용된 추론 강도만 검증한다.
- 임의 URL, renderer 제공 헤더, renderer 제공 계정 ID를 허용하지 않는다.

## 검증 기준

1. 새로고침 호출 한 번이 원격 모델 요청 한 번만 발생시킨다.
2. `gpt-5.6-terra`가 포함된 정상 응답은 모델 선택 목록에 Terra와 서버 제공 추론 강도를 표시한다.
3. 원격 모델을 선택한 뒤 실제 스트림 요청이 같은 모델 ID를 사용한다.
4. 인증 헤더에는 bearer 토큰과 계정 ID가 포함되지만 renderer 결과와 로그에는 노출되지 않는다.
5. 잘못된 응답, 빈 목록, HTTP 오류에서는 마지막 정상 목록을 유지하고 오류를 반환한다.
6. 관련 Vitest와 stage-tamagotchi 타입 검사가 통과한다.

## 범위 밖

- 모델 목록의 디스크 캐시와 ETag 재검증.
- 백그라운드 또는 주기적 자동 갱신.
- Codex CLI나 app-server 재도입.
- 모델별 가격 표시와 사용량 제한 예측.
- Codex 이외 공급자의 모델 조회 변경.
